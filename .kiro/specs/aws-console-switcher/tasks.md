# Implementation Plan

本計画は承認済みの [requirements.md](./requirements.md)（v6）と [design.md](./design.md) を実装作業へ落とし込むものである。要件 ID は design.md「Requirements Traceability」の正準 ID（3.1, 3.2, 3.2.1, 3.3, 3.4, 3.5, 4.1, 4.1.1, 4.1.2, 4.2, 5）を用いる。`(P)` は並行実行可能なタスクを示す。

- [ ] 1. プロジェクト基盤と共有契約のセットアップ
- [ ] 1.1 MV3 拡張プロジェクトの初期化とビルド構成
  - TypeScript strict モードでの拡張プロジェクトを初期化し、Service Worker / Content Scripts / Popup のエントリを分離してビルドできる構成を用意する
  - リモートコード（`eval`）を排除する厳格な CSP を前提にしたバンドル設定とし、拡張 ID を固定するための `key` フィールド運用を組み込む
  - `bw serve` 代替経路用の `http://localhost:8087` host_permission を本番ビルドから除外するビルド時フラグ機構を用意する
  - _Requirements: 4.1_
- [ ] 1.2 共有型とメッセージ契約の定義
  - 全ポート境界で用いる結果型（成功/型付きエラーの判別共用体）と、失敗 3 分類（前提条件・AWS 認証・DOM タイムアウト）およびエラーコード列挙を網羅的に定義する
  - 非秘匿メタデータ、セッション記録、フロー状態、拡張設定、セレクタ集合のデータ構造を定義する（パスワード・TOTP を保持しないことを型レベルで保証）
  - 拡張内メッセージ（Popup/Content Script → Service Worker）と Native Messaging プロトコル（要求・応答）の判別共用体を定義し、全要求が `requestId` を持つことを契約化する
  - _Requirements: 4.2, 5, 3.3_

- [ ] 2. Native Messaging ホスト（bw CLI ラッパー）
  - 独立した別プロセス境界。1.2 の契約確定後、拡張側タスク（4〜7）と並行して実装可能
- [ ] 2.1 (P) ホストプロセスと Native Messaging stdio プロトコルの実装
  - stdin/stdout のメッセージパッシングで拡張と通信するホストプロセスの骨格を実装し、ネットワークポートを一切公開しない
  - ホスト manifest の `allowed_origins` に本拡張 ID のみを登録し、他からの到達を構造的に遮断する
  - 全要求・応答を `requestId` で対応付け、共有ポート上の並行要求を取り違えないディスパッチ機構を実装する
  - 1.2 の契約にのみ依存し、拡張側実装と並行して進められる
  - _Requirements: 3.3, 4.1_
- [ ] 2.2 Vault アンロック・ロック・状態取得と BW_SESSION 保持
  - マスターパスワードを環境変数経由で受け渡し、TTY 非対話環境でアンロックを実行して得たセッション鍵を自プロセスのみで保持する
  - ロック要求で保持中のセッション鍵を破棄し、状態取得でアンロック状態と最終利用時刻を返す
  - マスターパスワードは受け渡し後ただちに破棄し、永続化・ログ出力を禁止する
  - _Requirements: 4.1.1_
- [ ] 2.3 フォルダ／アイテム列挙とシークレット取得のラップ
  - フォルダ一覧とフォルダ内アイテム一覧を取得し、非秘匿メタデータとして返す
  - アイテム取得はユーザー名・パスワードのみを返し、TOTP シードは応答に含めない（揮発性の徹底）
  - TOTP 取得は生成済み 6 桁コードと有効残秒数を返す
  - 未アンロックは前提条件エラー、未ログインは対応するエラーコードを返し、真のオブジェクト欠落（アイテム削除・無効 UUID）と一時的前提条件エラーを区別できるエラーを返す
  - _Requirements: 3.3, 3.4_
- [ ] 2.4 アイドル自動ロックタイマーと設定受領
  - 各要求受信時に最終利用時刻を更新し、プロセス常駐タイマーで設定済みアイドル時間の経過を判定して自動ロックを呼ぶ（CLI セッションは自動失効しないため必須）
  - 拡張から受領する設定（アイドルロック分数・TOTP 最小残秒数）を適用する唯一の伝達経路を実装する
  - アイドル自動ロックにより MFA 実効退化をアンロック窓に限定する方針を反映する
  - _Requirements: 4.1.1, 4.1.2_
- [ ] 2.5 ホスト側 TOTP 待機制御
  - TOTP 取得時に残秒数が設定閾値未満なら、次ローテーションまでホスト側で待機して新コードを返し、MV3 Service Worker のタイマー制約を回避する
  - AWS 側のコード再利用拒否・時刻ずれによる失効を避けるための残秒数保証を実装する
  - _Requirements: 3.3, 3.5_

- [ ] 3. シークレット取得ポート（アダプタ／プロバイダ抽象）
- [ ] 3.1 Native Messaging アダプタの実装
  - 単一共有ポート上で `requestId` をキーにした要求キューを管理し、応答を正しい要求へ demux する
  - `requestId` は暗号論的乱数で要求ごとに生成し、同時実行中の複数フロー間での衝突を実用上無視できるレベルまで低減する
  - ポート切断（ホストクラッシュ）時に保留中の全要求を切断エラーで即時 reject する
  - 取得経路を差し替え可能なアダプタとして抽象化し、将来の経路変更に備える
  - _Requirements: 3.3, 4.2_
- [ ] 3.2 Bitwarden 用クレデンシャルプロバイダの実装
  - アカウント一覧・認証情報・TOTP 取得の抽象を通じてアダプタ経由で Vault からオンデマンドに取得する実装を用意する
  - 将来の SSO 移行時に別実装を注入できるよう、中核が本抽象にのみ依存する構造とする
  - _Requirements: 4.2, 3.3_

- [ ] 4. Service Worker（ステートレス・プロキシとフロー調停）
- [ ] 4.1 メッセージルーティングとフロー状態の永続化
  - Popup・Content Script からのメッセージを判別共用体で受理し対応する処理へ振り分ける
  - 揮発状態をメモリに保持せず、フロー状態をタブ ID をキーにブラウザローカルストレージへ永続化して再起動耐性を持たせる
  - 応答完了後にマスターパスワード・パスワード・TOTP をメモリに残さない不変条件を保証する
  - _Requirements: 3.2, 5, 4.1_
- [ ] 4.2 ログイン自動化ステートマシンの実装
  - ルーティング → アカウント ID（条件付き）→ 認証情報 → MFA（条件付き）→ 完了/失敗の状態遷移を実装する
  - 認証情報待機状態を二重監視し、MFA 画面 DOM 検知は Content Script、コンソールへのリダイレクト検知は Service Worker が担う
  - MFA 未設定アカウントはリダイレクトで直接完了へ遷移させ、失敗扱いにしない
  - ルーティングを含む各状態からのユーザーキャンセルと、失敗状態からの再試行による初期状態復帰を実装する
  - _Requirements: 3.2, 3.5, 5_
- [ ] 4.3 グローバルタブ監視とクリーンアップ単一経路
  - タブ更新リスナーをグローバルスコープで登録し、対象タブがコンソール URL へ遷移したら完了遷移を駆動する
  - タブ削除リスナーをグローバルスコープで登録し、フロー進行中タブの閉鎖時にフロー状態と対応アラームを即時解消する
  - 完了・失敗・キャンセル・タブ閉鎖のクリーンアップをフロー状態削除とアラーム解除の単一経路に集約し、後続フローに陳腐状態を残さない
  - _Requirements: 3.2, 5_
- [ ] 4.4 アラームによるタイムアウト計測と MFA 再試行回復
  - フロー固有名のアラームを状態遷移ごとに登録し、状態依存の監視窓（通常ステップと MFA 待機で異なる延長窓）を適用する
  - Service Worker が終了していてもアラーム発火で起床し、アラーム名からタブ ID を解析して該当フローを復元する
  - MFA 待機の延長窓満了時、フォーム存続かつ再試行回数が上限未満の場合のみ TOTP を再発行し、再試行回数を永続化して再起動を跨いで上限を判定する
  - 上限超過・フォーム消失・観測不能はタイムアウトとして失敗へ遷移させる
  - _Requirements: 3.5, 3.2, 5_
- [ ] 4.5 アンロックフローと設定伝達
  - Popup から受け取ったマスターパスワードをホストへ受け渡すアンロック経路を実装し、受け渡し後ただちに破棄する
  - アンロック成功直後および設定変更時にホストへ設定（アイドルロック分数・TOTP 最小残秒数）を伝達する
  - _Requirements: 4.1.1_

- [ ] 5. Content Scripts（signin DOM 操作とコンソール状態検出）
- [ ] 5.1 セレクタ集合と順序付きフォールバック／動的更新機構
  - サインイン各ステップと認証エラー・コンソール検出のセレクタを順序付きフォールバックで適用する仕組みを実装する
  - 同梱既定値と設定上書きをバージョン比較し新しい方を採用する動的更新機構を実装する
  - _Requirements: 5, 3.2_
- [ ] 5.2 (P) サインイン DOM 検知・値注入・認証エラー検知
  - サインイン系ページで対象 DOM を検知し、アカウント ID／認証情報／TOTP を注入して送信する（SPA 的遷移は変化監視で補完）
  - アカウント ID 入力欄の有無で汎用エントリと Cookie 記憶済みを判別し、Cookie 記憶時は認証情報入力から開始する
  - 認証失敗マーカーを監視して認証エラーイベントを送出し、DOM 検知イベントには常にアカウント識別子を含める
  - 高速なプログラム的送信による CAPTCHA 誘発を検知したら手動介入へフォールバックする
  - 5.1 のセレクタ集合に依存。コンソール検出（5.3）とは別ファイルのため並行実装可能
  - _Requirements: 3.2, 3.5_
- [ ] 5.3 (P) コンソールタブの現ログイン状態検出
  - 静的注入対象外のコンソールページに対し、Service Worker が対象タブを判定して状態検出スクリプトを動的注入する経路を実装する
  - 検出した現ログイン識別情報でローカル記録を補正し、確証が得られない場合は控えめに「不確定」表示できる情報を返す
  - 5.1 のセレクタ集合に依存。サインイン処理（5.2）とは別ファイルのため並行実装可能
  - _Requirements: 3.1_

- [ ] 6. セッションマネージャ（複数セッション併存）
- [ ] 6.1 セッション前面化とタブ無効時フォールバック
  - 既にサインイン済みのセッションを対象タブの活性化とウィンドウフォーカスで前面化する
  - 対象タブが閉鎖済み等で無効な場合は新規ログインへフォールバックする
  - _Requirements: 3.2, 3.2.1, 4.2_
- [ ] 6.2 LRU 退避と最終アクセス時刻の追跡
  - 新規ログイン確定直前に同期的に上限判定を行い、同時 5 セッション超過時のみ最古セッションを 1 件退避してから追加する（一時的にも上限を超えない）
  - 退避はセッション記録の削除にとどめ、AWS 側セッションのサインアウト DOM 操作は行わない
  - ログイン完了・前面化・ユーザーのタブ活性化を反映して最終アクセス時刻を更新し、経由外利用が退避対象になるのを防ぐ
  - _Requirements: 3.2.1_

- [ ] 7. Popup UI
- [ ] 7.1 アカウント一覧・検索・状態表示
  - 切り替え可能なアカウント（エイリアス・アカウント ID・IAM ユーザー名）を一覧表示し、インクリメンタルサーチで即座に絞り込む
  - 現在ログイン中のアカウントを視覚的に区別し、確証が得られない状態は控えめに表示して誤った「ログイン済み」表示を避ける
  - _Requirements: 3.1_
- [ ] 7.2 アンロック入力・再試行/キャンセル操作・エラー通知
  - マスターパスワードの一時入力欄を実装し、送信後ただちに入力欄をクリアする
  - 待機状態のキャンセルと失敗状態からの再試行の操作を提供する
  - 失敗 3 分類に応じた行動可能な通知（ホスト起動/アンロック誘導、認証エラー通知、手動ログイン誘導）を表示する
  - _Requirements: 3.1, 3.5, 4.1.1_

- [ ] 8. メタデータキャッシュと UUID 再同期
- [ ] 8.1 非秘匿メタデータのキャッシュと初回同期
  - 設定のフォルダ名からフォルダ ID を解決してキャッシュし、フォルダ内アイテムを列挙して非秘匿メタデータ（UUID・アカウント ID・エイリアス・表示用ユーザー名・MFA 有無）をローカルにキャッシュする
  - Vault アンロック解除後の初回アクセス時に同期を再実行する
  - パスワード・TOTP シードはキャッシュしない
  - _Requirements: 3.4, 3.1, 3.3_
- [ ] 8.2 UUID 再同期トリガーの分類と自動ログイン制御
  - 真のオブジェクト欠落（アイテム削除・無効 UUID）のみ当該 UUID を即時無効化し、再同期完了まで該当アカウントの自動ログインを停止して通知する
  - 一時的前提条件エラー（Vault ロック・ホスト未起動等）ではキャッシュを保持し、無効化・再同期を行わず解消後に再試行できるよう誘導する
  - 手動同期操作とアンロック解除後初回アクセスで再列挙してキャッシュを上書き更新する
  - _Requirements: 3.4_

- [ ] 9. エラーハンドリング統合とセキュリティ最終化
- [ ] 9.1 失敗分類から UX への統合
  - 前提条件エラー・AWS 認証エラー・DOM タイムアウトの各コードに対する応答とフォールバックを統合する
  - TOTP 拒否は残秒数を保証した次コードで 1 回のみ自動再試行し、上限超過・認証エラーはフローを停止して通知する
  - 失敗を構造化ログとして記録し、秘匿値を出力しない
  - _Requirements: 3.5, 4.1.1_
- [ ] 9.2 manifest 権限最小化・CSP・秘匿境界の最終化
  - 権限を最小構成（Native Messaging・ストレージ・タブ・スクリプティング・アラーム）に限定し、Cookie 権限を含めない
  - host_permissions をサインイン系とコンソール用に限定し、`bw serve` 代替用の localhost は本番ビルドから除外する
  - Popup からの localhost 直接通信を禁止し必ず Service Worker 経由とする厳格 CSP を適用する
  - 秘匿値（セッション鍵・マスターパスワード）を拡張・ストレージへ保存しない境界を最終検証する
  - _Requirements: 4.1, 4.1.2, 4.2_

- [ ] 10. テスト
- [ ] 10.1 (P) ユニットテスト
  - ステートマシン遷移（各分岐＋MFA 有無＋キャンセル/再試行）、メッセージルーティング、`requestId` demux、セレクタフォールバック選択、UUID 再同期トリガー分類、フロー状態復元を検証する
  - 統合テスト（10.2）とは別テストスイートのため並行実施可能
  - _Requirements: 3.2, 3.4, 3.5, 5_
- [ ] 10.2 (P) 統合テスト
  - Native Messaging 往復（`allowed_origins` 制限・`requestId`・ポート切断）、bw ラップ（取得・列挙）、アンロックとアイドル自動ロック、ホスト側 TOTP 待機、ローカルストレージ補正反映を検証する
  - ユニットテスト（10.1）とは別テストスイートのため並行実施可能
  - _Requirements: 3.3, 3.4, 4.1.1_
- [ ] 10.3 E2E テスト（PoC 連動）
  - Native Messaging 疎通、アンロック＋アイドルロック、Service Worker 強制終了下でのフロー状態復元、AWS サインイン DOM ＋複数セッション UI、セレクタ耐性、TOTP シード形式確認を検証する
  - 全実装タスクの完了に依存
  - _Requirements: 3.2, 3.2.1, 4.1.1, 5_
